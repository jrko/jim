# Brainstorm: Secure skill / agent for jim

*2026-04-11*

## Seed

Extend jim with a `secure` capability — possibly a skill, possibly a new agent — to perform security analysis on specs and plans. In practice this has proven useful for catching design gaps and architectural issues *before* implementation.

## Shape: agent + skill

Leaning agent + skill. Security analysis is stateful domain reasoning — it needs to understand the project's threat landscape, data classification, trust boundaries. Similar to how `@jim:architect` owns architectural reasoning.

## Dual-lens model

Available any time (like all jim commands), but adapts its lens based on what it's reviewing:

**Spec-phase lens — "what's missing from the requirements?"**
- Data classification gaps (PII stored without retention/deletion policy)
- Missing security requirements (no rate limiting on user-facing endpoint, no authZ model specified)
- Unaddressed trust boundaries (who can access what, under what conditions?)
- Privacy and compliance omissions
- Threat surface identification

**Plan-phase lens — "how does the design handle it?"**
- Flawed mitigations (JWT without token revocation strategy)
- Privilege issues (migration runs as superuser, doesn't drop privileges)
- Trust boundary validation between components
- Crypto choices and key management
- Error handling that leaks information
- Dependency risk

Spec-lens catches omissions. Plan-lens catches flawed mitigations.

## Output: security.md

Findings written to `security.md` as a sibling artifact in the spec directory, alongside `spec.md`, `research.md`, `plan.md`. Gains:

- Institutional memory — persists as part of the spec record
- Referenceable by plan ("see security.md for threat model")
- Findings categorized and triageable
- Consumable as input by architect and PM on re-runs

### Feedback routing

At end of review, offer routing (like brainstorm does):
- "These findings should feed back into the spec" → user re-runs /jim:spec
- "These should modify the plan" → user re-runs /jim:plan
- "These are future hardening" → route to /jim:backlog

User decides what goes where. /jim:secure is advisory, not blocking.

## Security framework approaches considered

| Approach | What it does | Strengths | Weaknesses |
|----------|-------------|-----------|------------|
| **STRIDE** | Categorizes: Spoofing, Tampering, Repudiation, Info Disclosure, DoS, Elevation | Systematic, broad coverage, well-known | Can feel mechanical; may generate noise |
| **Attack trees** | Attacker goals modeled as trees of sub-goals | Focused, shows attack chains | Needs clear attacker model; harder to automate |
| **OWASP ASVS** | Verification checklist by level (L1/L2/L3) | Concrete, actionable | Web-app focused; overkill for internal tools |
| **Data-flow + trust boundaries** | Data flows with trust boundary markers | Visual, catches boundary-crossing naturally | Needs clear architecture; less useful at spec phase |
| **Freeform expert review** | LLM as experienced security reviewer | Adaptive, catches non-obvious issues, low ceremony | Inconsistent coverage |
| **Hybrid: freeform + STRIDE sweep** | Expert review first, then STRIDE as completeness check | Creative insights AND systematic coverage | Slightly more complex |

**Leaning hybrid.** Freeform expert review catches the non-obvious, context-specific issues. STRIDE sweep after ensures systematic coverage — it's general-purpose (not web-only like OWASP) and maps to architectural decisions in specs/plans.

## Suggested amendments

Findings should include concrete suggestions, not just flags. Example:

> **Finding:** No data retention policy for user session data.
> **Severity:** Notable
> **Suggested spec addition:** "User session data MUST be purged after 90 days of inactivity."
> **Route:** Spec amendment

Gives the PM or architect something actionable to accept, reject, or modify.

## Severity model

Three levels:

- **Critical** — Design flaw that will create a vulnerability if built as-is. Must be addressed before implementation.
- **Notable** — Gap that should be addressed before build. Missing controls, underspecified trust boundaries, incomplete threat coverage.
- **Advisory** — Hardening opportunity. Not a flaw, but worth considering. Good candidate for backlog routing.

Severity informs routing: critical → likely spec/plan rework; notable → spec/plan amendment or plan task; advisory → backlog.

## Scope boundary

This is a **design-time** review tool. It reviews specs and plans, not code. Post-build security scanning is out of scope — that's the domain of the planned `/jim:review` skill, which can call `@jim:security` for a post-build security lens.

## Architecture awareness

The security agent should read `ARCHITECTURE.md` (and potentially `VISION.md`) to ground its analysis in:
- Existing trust boundaries and component relationships
- Data flows and storage patterns
- Authentication/authorization patterns already in place
- Infrastructure and deployment context

This prevents findings that contradict or duplicate existing architectural decisions.

## security.md lifecycle

**Advisory, not a gate.** Consistent with jim's human-in-the-loop philosophy — the human decides what to act on.

**Edit-in-place, like spec.md and plan.md.** Re-running `/jim:sec` does a differential update — reads existing security.md, updates findings to reflect current state. Git is the audit trail, same as every other spec artifact. Keeps symmetry with how the rest of the workflow operates.

**Graceful with missing artifacts.** If run when spec exists but plan doesn't (or vice versa), note the absence and proceed with what's available.

No hard gates on `/jim:plan` or `/jim:build` — security.md is a tool for informed decision-making, not a blocker.

## Naming

Skill: `/jim:sec`

Agent name candidates:

- **`@jim:sec`** — Matches the skill name. Short, consistent. But "sec" as an agent name feels a bit terse — agents tend to be nouns (pm, architect, coder, researcher).
- **`@jim:security`** — Clear, professional. Slightly longer but reads naturally: "the security agent reviewed your plan."
- **`@jim:sentinel`** — Evocative — a sentinel watches for threats. Distinctive from the other role-based names. But breaks the convention of agents being named after roles.
- **`@jim:auditor`** — Role-based, describes what it does. But "audit" implies compliance/checkbox work, which is broader than the intent.
- **`@jim:secarch`** — Security architect. Descriptive but awkward to say.
- **`@jim:threatmod`** — Too specific; the agent does more than just threat modeling.

**Decision:** Skill = `/jim:sec`, Agent = `@jim:security`

## Researcher integration

The secure agent should be able to spawn `@jim:researcher` for questions it can't answer from context alone:
- "Is this crypto library still considered safe?"
- "What are the known attack vectors for this auth pattern?"
- "What does the compliance landscape look like for this data type?"

Mirrors how `@jim:architect` already calls on the researcher. The secure agent focuses on *identifying* what needs investigation; the researcher does the legwork.
